# 執筆規約（全セクション共通・必読）

このカリキュラムは「プログラミングスクールの教材」である。読者はプログラミング初学者（入門編でHTML/CSS/JS/TypeScriptを修了済み）。半年でCI/CD・コンテナ・データベースを理解し、SNSアプリを一人で作れるようになることがゴール。

## 文体・トーン

- です・ます調。教科書として成立する落ち着いた文体。
- 「初心者の皆さん」「〜ですよね！」「頑張りましょう！」のような呼びかけ・煽り文体は禁止。
- 専門用語は初出時に必ず定義する。英語の用語には読み（カタカナ）を添える。
- 概念の説明 → 具体例 → 手を動かす、の順で構成する。
- 「なぜそうするのか」を必ず説明する。手順の羅列にしない。

## ページのfrontmatter

```yaml
---
layout: default
title: <ページタイトル>
parent: <セクションのtitle（index.mdのtitleと完全一致させること）>
nav_order: <セクション内の連番>
---
```

セクションの index.md は:

```yaml
---
layout: default
title: <セクションタイトル>
nav_order: <toc.mdで指定された番号>
has_children: true
permalink: /<ディレクトリ名>/
---
```

## ページの標準構成（この順序を守る）

1. `# タイトル` + 導入段落（このページで学ぶこと・前のページとのつながり）
2. `## 学習目標` — 箇条書き3〜5個
3. 本文セクション（複数の `##`）
4. `## 理解度チェック` — 3〜6問。形式は下記。
5. `## セルフレビュー` — `- [ ]` 形式のチェックリスト5〜8個（「〜を自分の言葉で説明できる」「〜を写経せずに書ける」など）
6. `## 次のステップ` — 前後ページへのリンクと、このページの内容が後でどこで使われるかの予告

### 理解度チェックの形式

```markdown
**Q1. 質問文**

<details markdown="1">
<summary>解答を見る</summary>

解答と解説。なぜそうなるのかまで書く。

</details>
```

## コード

- すべてのコードは**完全で動くもの**を書く。断片の場合はどのファイルのどこに足すか明示する。
- 重要なコードブロックの直後に「**コード解説**」として1行ごと（または意味のあるまとまりごと）の解説を箇条書きで入れる。初学者が読み飛ばせる程度に簡潔に、しかし省略しない。
- ファイル名はコードブロックの直前に `**`src/main.ts`**` のように太字で示す。
- ターミナルコマンドは実行結果の例も載せる。
- バージョンを固定: Node.js 20 / TypeScript 5系 / React 18 + Vite 5 / NestJS 10 / Prisma 5 / PostgreSQL 16 / Jest 29。
- **パッケージマネージャは pnpm に統一**（npm/yarnは使わない）。導入は `corepack enable pnpm`（Node.js 20に同梱のCorepackを使う。環境構築章で案内済みでない点に注意し、初出のセクションでは導入コマンドと「pnpmとは何か・npmとの違い（ディスク効率・速度）」の短い説明を入れる）。
  - インストール: `pnpm install` / 追加: `pnpm add <pkg>`（開発依存は `pnpm add -D`）
  - スクリプト実行: `pnpm run dev` など / 一時実行（npx相当）: `pnpm dlx <pkg>`（例: `pnpm dlx create-vite`）
  - CIでは `pnpm/action-setup` + `actions/setup-node` の `cache: pnpm` を使う。
- **開発環境の標準形**: データベース（PostgreSQL）はdocker-composeで起動し、APIとフロントエンドはローカルで `pnpm run dev` で起動する。アプリ自体のコンテナ化はデプロイ（本番）のために学ぶ、という整理で一貫させる。

## 図（Mermaid）

- 概念を説明するページには最低1つMermaid図を入れる。通信や処理の流れには**シーケンス図**を必ず使う。
- ` ```mermaid ` コードブロックで書く（just-the-docsが描画する）。
- 配色は初学者が見やすいように: ノードの強調には `style ノードID fill:#e3f2fd,stroke:#1565c0`（青系=主役）、`fill:#e8f5e9,stroke:#2e7d32`（緑系=成功/結果）、`fill:#fff3e0,stroke:#ef6c00`（橙系=注意/外部）を使う。赤背景は使わない。
- 図には必ず前後に説明文を付ける。図だけで説明を終わらせない。

## ページ間参照（重要）

- 過去に学んだ内容に触れるときは必ずリンクを張る: `[JavaScript基礎](../frontend/javascript_basics.html)` のように**相対パス + `.html`**で書く（セクションindexへは `../react/` のようにディレクトリパスで書く）。
- 各ページのURLは `toc.md` に列挙されている。リンク先はtoc.mdに存在するパスのみ使うこと。
- 特にSNS開発セクションでは「→ 詳細は[認証の基礎](../backend/...)を参照」のように過去ページへの参照を多用する。

## 量の目安

- 通常ページ: 300〜600行のMarkdown。
- 実践・構築ページ: 500〜900行。
- testing セクションのみ意図的にコンパクト（各ページ250〜400行）。

## してはいけないこと

- 嘘のAPI・存在しないオプションを書かない。不確かな細部は一般的で安定した書き方に寄せる。
- AWSの章では**料金が発生する操作には必ず注意書き**を入れ、無料利用枠と削除手順に言及する。
- 外部教材へ丸投げしない（リンクは補足として可）。本文だけで完結させる。
